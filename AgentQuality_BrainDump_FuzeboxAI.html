2. Three Framework Pillars

The core insight: we are not inventing a new framework. We are applying three existing, citable, peer-reviewed frameworks to a problem nobody has solved yet — measuring AI agent quality in the language business leaders already speak.

"We didn't invent a new metric. We applied three established frameworks — used in manufacturing, quality management, and AI governance — to agent measurement. For the first time in one place."

ASQ / ISO 13053

Six Sigma — DPMO

Defects per million opportunities mapped to agent invocations. Defect = task failure, latency P90 breach, or cost overrun. Universal quality language enterprises already use.

Cite: American Society for Quality (ASQ), IEC 60812 for FMEA

ISO 22400 — non-hospitality

OEE — Overall Equipment Effectiveness

Availability × Performance × Quality. Maps directly: agent uptime × latency SLA × task success rate. 85%+ = world class. Every ops leader already knows this number.

Cite: ISO 22400-2, adopted in logistics, BPO, and tech ops

Parasuraman et al. — hospitality only

SERVQUAL — Service Quality

Reliability, responsiveness, assurance, empathy, tangibles. Replaces OEE when client is hotel, casino, or hospitality. Same underlying calculation, different vocabulary.

Cite: Parasuraman, Zeithaml & Berry (1988) — widely cited in service ops literature

ISO/IEC 42001:2023

AI Governance Layer

AI management system standard. Audit logs, human oversight, decision traceability. EU AI Act Article 14 (Aug 2025) compliance anchor. Gives compliance team something to cite.

Cite: ISO/IEC 42001:2023, EU AI Act Article 14 on human oversight

O*NET OnLine / BLS

Human Labor Backbone

Task taxonomy, time weights, and automation susceptibility for every US occupation. Turns the human side of the equation from assumption into citable public data.

Cite: O*NET OnLine (onetcenter.org), BLS Occupational Outlook Handbook

Google SRE / IEEE P2675

SLO / Error Budget

Service Level Objectives — universally understood in tech orgs. Error budget = how much unreliability is acceptable. Bridges Six Sigma language to engineering teams.

Cite: Google SRE Book (free), IEEE P2675 DevOps standard

4. O*NET Connection — Human Labor to ROI

O*NET is the connective tissue that ties all three of Les's asks together. It doesn't just list job titles — for every occupation it provides tasks with time weights, skills, and automation susceptibility scores. This means the human side of the equation is not invented; it is traceable to a US government public database.

Data Flow

O*NET API

Task list + time weights + automation score

→

Coverage mapper

Agent / collaborative / human per task

→

BLS wage data

Hourly cost of human per task (public)

→

Langfuse telemetry

Agent cost + latency + success rate

→

Symmetry equation

Net ROI, honest and auditable

Three O*NET Connection Points

  • Connection to Labor Graph — Pull O*NET task list for target occupation. Each task gets one of three states: Agent Owned, Collaborative (agent does, human approves), or Human Retained. The before/after labor graph is this task list reshuffled between those three buckets as agent quality improves.
  • Connection to Symmetry Equation — O*NET has time allocation weights per task. When a task moves from human to agent, you calculate hours freed (O*NET time weight × BLS wage) minus agent cost (Langfuse telemetry) minus oversight overhead. This is the honest ROI correction: 23-25% gross saving corrected to ~10% net.
  • Connection to Governance Audit Trail — O*NET's collaborative tasks map directly to the EU AI Act's human oversight requirement. Every task in the collaborative bucket needs an audit log entry: who reviewed, what they decided, how long it took, what % of decisions they overrode. This turns compliance into a measurable process metric.

Worked ROI Example

Occupation: Sports Betting Analyst | O*NET returns: 23 tasks, avg 40hrs/week, $42/hr BLS wage | Agent covers: 8 of 23 tasks at 3.8σ quality | Those 8 tasks = 35% of weekly time = 14hrs = $588/week/person | Agent cost for same tasks = $0.022 per run × 200 runs/week = $4.40 | Gross saving = $583.60/week/person | Minus oversight: 2hrs/week human validation = $84 | Minus governance overhead: 0.5hrs = $21 | NET SAVING = $478.60/week/person

This is a credible, auditable number. Not 83,000% ROI. A CFO-defensible figure with every assumption visible and traceable to public data sources.

6. Open Questions Before Build

Four decisions needed before the Claude Code prompt is written. Each one has architecture implications.

Langfuse connection

Direct API read or pre-processed batch? Real-time vs batch affects the proxy latency question raised in the call. Direct is more impressive; batch is more stable for April 7.

O*NET for April 7

Live API calls to onetcenter.org (requires free account registration) or 2-3 seeded occupations in mock data? Live is more impressive; seeded is safer for demo stability.

Industry flag

Where in the setup flow does the OEE vs SERVQUAL switch happen? This is a UI flag, not a data architecture change. Needs an industry selector dropdown at process setup screen.

Client benchmark input

Does the client input their own quality target (e.g. 'we operate at 4σ minimum') or does the platform use a sensible default? Client-set target makes the governance story significantly stronger.
